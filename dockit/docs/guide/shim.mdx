# Shim 机制

PVM 的核心设计是 **shim（垫片）驱动**——它不需要修改系统 PATH 去指向具体版本目录，
而是在 `~/.pvm/shims` 下为每个可执行命令生成一个 shim。

## 原理

```
~/.pvm/shims/node      ← shim（pvm 本体的硬链接，在 PATH 最前面）
        │
        ▼
pvm 解析版本 → 找到 ~/.pvm/installs/<rt>/<ver>/...（或 versions 指向的目录）
        │
        ▼
执行真正的二进制
```

shim 的本质是 **pvm 自身的硬链接**（而非独立小脚本或独立二进制）。pvm 启动时检测
自身文件名（`node` → node 运行时，`git` → git 运行时），按文件名自分发到对应
的真实版本。

- 安装新版本后，PVM 调用 `reshim` 重新生成 shims，使新命令可被 PATH 发现。
- 切换用户/项目版本时，shim 在运行时解析目标版本，**无需改动 PATH**。
- 硬链接**零磁盘占用**：28 个 shim 与 pvm 本体共享同一个 inode，总占用即一份 pvm 的大小。

## 单二进制设计

PVM 是**单一二进制**——`pvm`（或 `pvm.exe`）既是管理工具，也是 shim 源。
`reshim` 把 `~/.pvm/bin/pvm` 硬链接为 `node`、`npm`、`git`、`python` 等命令名：

```
~/.pvm/bin/pvm        ← 主程序（管理入口）
~/.pvm/shims/node     ← 硬链接（共享同一份内容）
~/.pvm/shims/npm      ← 硬链接
~/.pvm/shims/git      ← 硬链接
...
```

这样做的好处：下载一个文件就够了，不存在"忘了下载 shim 文件导致 setup 假成功"的问题。
跨平台（Windows / macOS / Linux）统一使用硬链接，硬链接不可用时（如跨文件系统）自动回退到复制。

## 为什么不用管理员权限

传统版本管理器常需要写入系统目录或改系统 PATH。PVM 的方案：

- `shims/` 与 `bin/` 都在用户目录 `~/.pvm/` 下。
- 只需把 `~/.pvm/shims` 加进**用户级** PATH。
- IDE（如 VSCode）通过 shims 自动识别 `node` / `git` 等命令。

:::tip 与 nvm 共存
如果系统 PATH 中已有 nvm 的 node（如 `D:\node`），`pvm setup` 会自动把 pvm shims
**前置到系统 PATH 最前**（需管理员），让 pvm 的版本优先于 nvm。详细见 [setup](/commands/setup)。
:::

## Windows 上的 bin 目录

`~/.pvm/bin/` 下会额外生成 `git.exe` / `bash.exe` 等文件（硬链接 pvm.exe 或 real exe 复制），
供 VSCode 等依赖 `.exe` 路径探测的工具使用。

## 重新生成 shims

```bash
pvm reshim
# 安装/卸载版本后，若命令未被识别，手动执行一次
```

:::tip 提示
误删或命令"装了找不到"时，优先跑 `pvm reshim`，而不是重装。
:::
